# ERP Suite 실무 투입 검증 기준서

## 1. 문서 개요

### 1.1 목적
본 문서는 ERP Suite 시스템의 실무 투입 전 안정성, 보안성, 데이터 무결성, 성능 및 기능 완전성을
체계적으로 검증하기 위한 기준을 정의한다. ISMS/ISO 27001 및 OWASP Top 10 기준에 준하여 작성하였다.

### 1.2 범위
- 대상 시스템: ERP Suite (Django 5.x 기반 제조/판매 통합 ERP)
- 대상 앱: core, accounts, inventory, production, sales, service, accounting, investment, warranty, marketplace, inquiry, purchase, attendance, board, calendar_app, hr, messenger
- 대상 환경: 개발(SQLite), 운영(PostgreSQL + Docker)

### 1.3 검증 주기
| 구분 | 주기 | 비고 |
|------|------|------|
| 보안 검증 | 분기 1회 | 배포 전 필수 |
| 데이터 무결성 | 월 1회 | 배포 전 필수 |
| 성능 검증 | 분기 1회 | 대량 데이터 발생 시 수시 |
| 기능 워크플로우 | 배포 시마다 | CI/CD 파이프라인 통합 |
| 장애 복구 | 반기 1회 | 재해 복구 훈련 연계 |

### 1.4 담당자
| 역할 | 담당 | 비고 |
|------|------|------|
| 검증 책임자 | 시스템 관리자 (admin) | 최종 승인 |
| 검증 수행자 | 개발팀 | 자동화 테스트 실행 |
| 결과 검토자 | 매니저 (manager) | 결과 리뷰 |

## 2. 보안 검증 (OWASP Top 10 기반)

### SEC-001: SQL Injection 방어

| 항목 | 내용 |
|------|------|
| **검증 항목명** | SQL Injection 공격 방어 |
| **검증 기준** | 모든 DB 접근이 Django ORM 또는 파라미터화 쿼리를 통해 이루어지며, raw SQL 사용 시 파라미터 바인딩이 적용되어야 한다. `extra()`, `raw()` 사용 시 사용자 입력이 직접 삽입되지 않아야 한다. |
| **Pass 조건** | 소스코드에서 `raw()`, `extra()`, `cursor.execute()` 사용 시 모두 파라미터 바인딩 적용 확인. 악의적 입력(`' OR 1=1 --`)에 대해 정상 오류 반환. |
| **Fail 조건** | 문자열 포맷팅으로 SQL 쿼리를 구성하는 코드가 1건 이상 발견됨 |
| **검증 방법** | 자동 - 소스코드 정적 분석 (grep 기반) + 테스트 코드 |
| **위험 등급** | **상** |

### SEC-002: XSS(Cross-Site Scripting) 방어

| 항목 | 내용 |
|------|------|
| **검증 항목명** | XSS 공격 방어 |
| **검증 기준** | Django 템플릿 자동 이스케이프가 활성화되어 있어야 하며, `|safe` 또는 `{% autoescape off %}` 사용 시 정당한 사유가 있어야 한다. |
| **Pass 조건** | `|safe` 필터 사용이 0건이거나, 사용 시 입력 검증이 확인됨. 템플릿에서 자동 이스케이프가 비활성화되지 않음. |
| **Fail 조건** | 사용자 입력이 `|safe`로 출력되는 곳이 1건 이상 발견됨 |
| **검증 방법** | 자동 - 템플릿 파일 정적 분석 + 수동 리뷰 |
| **위험 등급** | **상** |
