# ERP Suite 실무 투입 검증 기준서

## 1. 문서 개요

### 1.1 목적
본 문서는 ERP Suite 시스템의 실무 투입 전 안정성, 보안성, 데이터 무결성,
성능, 기능 완전성을 체계적으로 검증하기 위한 기준을 정의합니다.
ISMS/ISO 27001 및 OWASP Top 10 표준에 따라 작성되었습니다.

### 1.2 범위
- **대상 시스템**: ERP Suite (Django 5.x 기반 제조/판매 통합 ERP + 그룹웨어)
- **대상 앱**: core, accounts, inventory, production, sales, service, accounting, investment, warranty, marketplace, inquiry, purchase, attendance, board, calendar_app, hr, messenger, ad
- **대상 환경**: 개발 (SQLite), 스테이징 (PostgreSQL), 운영 (PostgreSQL + Docker)

## 2. 보안 검증 (OWASP Top 10 기반)

### SEC-001: SQL Injection 방어

| 항목 | 내용 |
|------|------|
| **명칭** | SQL Injection 공격 방어 |
| **기준** | 모든 DB 접근은 Django ORM 또는 매개변수화된 쿼리를 사용해야 함. `raw()`, `extra()`, `cursor.execute()` 사용 시 반드시 파라미터 바인딩 적용. 사용자 입력이 SQL 문자열에 직접 삽입되어서는 안 됨. |
| **통과** | (1) 소스코드 정적 분석 결과 문자열 포맷팅 SQL이 0건. (2) 악의적 입력(`' OR 1=1 --`, `'; DROP TABLE--`, `UNION SELECT`)에 대해 적절한 에러 응답 반환. (3) 파라미터 바인딩 없는 `raw()` 또는 `extra()` 호출 0건. |
| **실패** | 사용자 제어 입력을 문자열 포맷팅/연결로 구성한 SQL 쿼리가 존재하는 경우 |
| **방법** | 자동화 - 정적 분석 (grep/semgrep) + 인젝션 페이로드 Django TestCase |
| **테스트 코드** | `tests/verification/test_security.py::TestSQLInjection` |
| **위험도** | **심각** |
| **OWASP 참조** | A03:2021 - Injection |

### SEC-002: XSS (Cross-Site Scripting) 방어

| 항목 | 내용 |
|------|------|
| **명칭** | XSS 공격 방어 |
| **기준** | Django 템플릿 자동 이스케이프가 활성화되어야 함. `|safe` 필터 또는 `{% autoescape off %}` 사용 시 문서화된 근거와 입력 검증 필요. 모든 사용자 생성 콘텐츠는 출력 시 이스케이프 처리. |
| **통과** | (1) `|safe` 사용 횟수가 0이거나 각 인스턴스에 검증된 입력이 있음. (2) 사용자 입력과 함께 `{% autoescape off %}` 미사용. (3) `<script>alert('xss')</script>` 입력이 이스케이프된 텍스트로 렌더링. (4) CSP 헤더가 인라인 스크립트 실행 차단. |
| **실패** | 사용자 입력이 살균 없이 `|safe`로 렌더링되거나, 저장형 XSS가 가능한 경우 |
| **방법** | 자동화 - 템플릿 정적 분석 + 수동 검토 + XSS 페이로드 인젝션 테스트 |
| **테스트 코드** | `tests/verification/test_security.py::TestXSS` |
| **위험도** | **심각** |
| **OWASP 참조** | A03:2021 - Injection |

### SEC-003: CSRF 토큰 검증

| 항목 | 내용 |
|------|------|
| **명칭** | CSRF 토큰 유효성 검증 |
| **기준** | 모든 POST/PUT/DELETE 폼에 `{% csrf_token %}`이 포함되어야 함. `CsrfViewMiddleware`가 활성화되어야 함. JWT 인증을 사용하는 API 엔드포인트 외에는 `@csrf_exempt` 사용 금지. |

| **실패** | POST 폼에 csrf_token 누락 또는 비-API 뷰에 `@csrf_exempt` 사용 |
| **방법** | 자동화 - CSRF 없는 POST 요청 테스트 + 템플릿 스캔 |
| **테스트 코드** | `tests/verification/test_security.py::TestCSRF` |
| **위험도** | **상** |
| **OWASP 참조** | A01:2021 - Broken Access Control |

| **기준** | 모든 비즈니스 뷰에 `LoginRequiredMixin` 또는 동등한 인증이 적용되어야 함. 미인증 사용자는 로그인 페이지로 리다이렉트. 인증 없이 비즈니스 데이터 접근 불가. |

| **실패** | 인증 없이 접근 가능한 비즈니스 URL이 존재하는 경우 |
| **방법** | 자동화 - URL 패턴 열거 + 익명 접근 테스트 |

### SEC-005: 권한 상승 방지 (RBAC)

| 항목 | 내용 |
|------|------|
| **명칭** | 역할 기반 접근 제어 검증 |
| **기준** | `AdminRequiredMixin` 뷰: admin 역할만 접근. `ManagerRequiredMixin` 뷰: manager/admin만 접근. staff 역할은 관리자 기능 접근 불가. 수평적 권한 상승(ID 조작으로 다른 사용자 데이터 접근) 차단 필수. |
| **통과** | (1) staff 계정이 admin 뷰 접근 시 → 403. (2) staff가 manager 뷰 접근 시 → 403. (3) manager가 admin 뷰 접근 시 → 403. (4) 사용자 A가 URL ID 파라미터 변경으로 사용자 B의 데이터 접근 불가. |
| **실패** | 하위 권한 사용자가 상위 권한 기능에 접근하거나, 수평적 접근이 가능한 경우 |
| **방법** | 자동화 - 역할 기반 접근 테스트 매트릭스 + IDOR 테스트 |
| **테스트 코드** | `tests/verification/test_security.py::TestRBAC` |

pip-audit
